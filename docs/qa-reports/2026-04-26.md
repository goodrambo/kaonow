## 🔴 Blockers（user 完全無法作答）

### B-1. `cap-112-math-010` — 4 個空選項 + 題幹混入選項片段

**現象**

```
題幹: "利用公式解可得一元二次方程式$3x^{2}$−11x−1=0的兩解為a、b，且a>b，求a值為何？
       −11+1096−+133(B)116+109(C)116+133(D)116"
選項: ["", "", "", ""]
ans:  3 (即 D)
exp:  "...故 (D) 正確" ← 詳解內容是對的
```

**推測根因**：PDF parser 把選項抽進題幹（題幹尾巴是 `(B)116+109(C)116+133(D)116` 是選項殘留），同時 `opts` 陣列被填入 4 個空字串。詳解和 ans 都正確，所以是「題目/選項抽取階段」的 parser bug，不是答案 bug。

**建議修法**：去 raw PDF 重抽這題，預期 4 選項分別是 `(11+√133)/6` 等公式解結果。可用 vision OCR fallback 或手抄。

**責任模組**：content pipeline / W1.9 cap parser

## 🟡 Majors

### M-1 ~ M-4. `KATEX_DOUBLE_ESC`（4 題，都在會考數學）

| id | 題幹片段 |
|---|---|
| `cap-104-math-001` | `算式 $(-1\\\\frac{1}{2}) \\\\times (-3\\\\frac{1}{4}) \\\\times \\\\frac{2}{3}$` |
| `cap-109-math-004` | `算式 $\\\\sqrt{2} \\\\times (\\\\sqrt{48} - \\\\sqrt{12})$` |
| `cap-112-math-004` | `化簡 $\\\\sqrt{135}$ 的結果` |
| `cap-114-math-008` | `計算 $(2\\\\sqrt{3}+\\\\sqrt{6})\\\\times\\\\sqrt{2}$` |

**推測根因**：和 [memory `postgres_sql_escape_rule.md`](../../memory/postgres_sql_escape_rule.md) 同一根因——標記式應為 `\frac` `\sqrt`，被存成 `\\frac` `\\sqrt`，KaTeX 不渲染就 fallback 顯示原文。memory 提到「2026-04-26 vision 4 題踩過」可能就是這 4 題、或者還沒 cover 到。

**建議修法**：直接 SQL 全量替換：
```sql
-- 預檢
SELECT id, question_text FROM questions
 WHERE exam_id = 'cap' AND question_text ~ '\\\\(frac|sqrt|sum|int|cdot|times)';
-- 修正（確認預檢結果後再跑）
UPDATE questions SET question_text = REPLACE(question_text, E'\\\\\\\\', E'\\\\')
 WHERE exam_id = 'cap' AND question_text ~ '\\\\(frac|sqrt|sum|int|cdot|times)';
-- options / explanation 同理
```

**責任模組**：content pipeline / 已知事故（見 memory）

### M-11. `PARENT_PASSAGE_MISSING`（454 題，跨所有閱讀題）⚠️ 待驗證

**現象**：454 題子題的 `parent_id` 指向 cache 內找不到的 passage record。例如：

| id | parent_id | 題幹 | media |
|---|---|---|---|
| `cap-104-chinese-034` | `cap-104-chinese-passage-34` | `根據本文，「我相信對那些有慧根的人...」這句話的涵義與下列何者最接近` | `null` |
| `cap-104-chinese-035` | `cap-104-chinese-passage-34` | `下列何者最可能是本文所要傳達的人生態度？` | `null` |

**全部 454 題 `media` 都是 `null`**——子題沒有 inline passage_text，passage 也不在同一份 cache。

**影響範圍**（按 paper 分組）：

| paper 系列 | 缺 passage 子題數 |
|---|---|
| 國文 (chinese) 104-114 | 14-17 題/年 ≈ 156 題 |
| 英語閱讀 (english-reading) 104-114 | 16-20 題/年 ≈ 196 題 |
| 自然 (nature) 104-114 | 4-7 題/年 ≈ 56 題 |
| 社會 (society) 104-114 | 1-5 題/年 ≈ 26 題 |

**為什麼標 ⚠️ 待驗證**：兩種可能性：
1. **真 bug**：passage parent record 沒被 ingest 進 `questions` 表，user 進去就看到「根據本文」但無本文（**user-blocking**）
2. **cache 設計**：passage 不存 `kaonow_q16_*` cache，前端 render 時另外去 fetch（如有 `loadPassage(parent_id)` 之類函式）

**驗證方法**（建議 Rambo 跑）：
```sql
-- 看 supabase 是否實際有 passage records
SELECT count(*) FROM questions WHERE exam_id='cap' AND id LIKE '%passage%';
-- 隨手挑一個子題的 parent_id 確認
SELECT id, type, question_text, media FROM questions WHERE id='cap-104-chinese-passage-34';
```

如果是 (1)：454 題實質都是 blocker，必須補 passage。
如果是 (2)：在 L1 script 裡把 PARENT_PASSAGE_MISSING 從 assertion 拿掉、改用 L2 visual 檢查 passage 是否真的渲染出來。

**責任模組**：question-bank-ingest / content-model

## L2 Visual 觀察（POC 期間順手記）

POC 跑 L2 流程時順手記下幾個 UI/UX 觀察，不分嚴重度：

1. **首頁「📍最近瀏覽 / 1 個考試 / 會考」卡片不可點** — user 預期能直接點進「會考」，但卡片本身沒有 click handler，要繞回「考試專區」清單才能進去。
2. **`#/catalog/edu` URL 不導航** — 鍵入合法形式的類別 slug 但畫面停在 catalog 主頁，無 404、無錯誤、無轉址。要嘛該支援、要嘛該明確 fallback。記憶提到正確 slug 不含 `edu`（是 `civil/prof-exam/...`），但 URL 使用者打錯時的 UX 仍待改進。
3. **exam 詳情頁「5 章節數」vs「7 個科目」** — 兩個數字並列且都在頂部，user 困惑哪個才對。實際上 7 個科目對應後面卡片數量，但「章節數=5」沒解釋根據。
4. **科目頁「立即開始」「開始練習」兩個 CTA 並列** — 「熟練度」卡片的 CTA 跟「今日建議」的 CTA 都導向開始作答，差異不明。
5. **進入 quiz 後第一題渲染前底部出現「載入題目中...」** — 但題目其實已經渲染了，這個 toast 屬於誤導。
6. **作答後詳解品質佳** — 第一題 `cap-104-chinese-?` 的詳解完整解釋了 A 對的理由 + 逐一說明 B/C/D 為何錯，這是 KaoNow 的核心競爭力。✅

L2 觀察沒有正式進 issue tracker，等 L2 機制成熟後再系統化。
